Terminal Terminal emulator the dashboard's spawn auto-focus / shell-attach opens. Empty = auto-detect.
Window focus tclaude normally stamps a tclaude:<id> title on each agent's terminal window/tab. That title is how it finds an agent's existing window to raise it (window focus) and to auto-tile. On a plain desktop terminal some find it ugly — uncheck to leave the terminal's own tab title alone. Trade-off: focus/tiling then can't locate the window, so "focus" falls back to opening a new window instead of raising the existing one (affects WSL and native-Linux/X11; the explicit "open window" action still works). Leave on for WSL, where window focus depends on it. On (default). Stored as focus.window_title (the default is omitted; unchecked writes false). When on, focusing an agent that has no open window does nothing, instead of opening a new terminal attached to it. Use a row's "open window" action to open one explicitly. Off (default) = open-on-focus. When on, using the 🪟 windows… modal / command palette to focus more than one agent's window rearranges them into a tidy layout instead of leaving them where the OS dropped them. All windows are gathered onto one monitor — the one the first window is on — so a multi-monitor setup isn't scattered. By default windows keep their current size and are only repositioned; tick resize windows to fill the screen for the older screen-filling grid. Best-effort per platform (macOS AppleScript, Linux xdotool/kdotool, WSL PowerShell); an unsupported desktop leaves windows as-is. Gap = pixels between tiles; margin = inset from the screen edges. A single focused window is never tiled. Off (default).
Pre-compact guard Refuses Claude Code's automatic compaction until used context reaches a per-window floor — so a 1M session doesn't compact at the 200K boundary (~20% of the 1M bar). Lets context grow (and you reincarnate) first. Fails open when usage is unknown. By default only auto-compaction is blocked, never a /compact you type yourself. Floors — for each context-window size (tokens), the minimum used context (tokens) required before compaction is allowed. Empty = built-in defaults (150k of 200k, 800k of 1M).
Resume-from-summary prompt Claude Code shows an interactive "Resume from summary" chooser when you resume a session that's both old and large — which hangs tclaude's scripted resume, since a detached pane can't answer it. These override the age / size thresholds CC uses to decide; the prompt fires only when both are exceeded, so raising either high enough suppresses it. tclaude setup --install-resume-threshold-override sets minutes to 525600000 (≈1000 years) to switch it off. Empty = Claude Code's defaults (70 min / 100k tokens); 0 always shows it. Injected as CLAUDE_CODE_RESUME_* env vars on tclaude-spawned claude panes only — never written to ~/.claude/settings.json, so your manual claude runs are untouched. Note: these are undocumented, version-specific Claude Code env vars (verified against CC 2.1.187), so tclaude treats them as best-effort — if a future Claude Code build renames or drops them, this quietly becomes a no-op rather than an error.
Transcript retention Claude Code deletes a conversation transcript (and other stale session data / orphaned worktrees) once it's been inactive this many days — swept at Claude Code startup. Claude Code's own default is 30 days, so transcripts you haven't touched in a month vanish. Raise this to keep them longer; set a large value like 99999 to effectively keep them forever (Claude Code rejects 0 and has no "never" option). Empty = leave Claude Code's default (or whatever you set by hand) untouched. Unlike the resume-threshold overrides above, this is written to ~/.claude/settings.json (as cleanupPeriodDays) on every session start, so it also protects transcripts from your own plain claude runs. Stored as claude_cleanup_period_days.
Record hooks Debugging aid — verbose; leave off for normal use.
TUI color scheme The color palette tclaude's interactive terminal views — tclaude session ls -w, conv ls -w, and agent inbox -w — render with. Default is tuned to stay readable on both light and dark terminals (a little dimmer on a dark background). Dark mode (high contrast) restores the brighter earlier palette (vivid yellow / green / red and a brighter header) for higher contrast on a dark terminal, at the cost of light-terminal readability. Applies to newly launched watch views. Stored as tui.color_scheme (the default is omitted).

Costs

Cost display multiplier Scales every displayed cost estimate — the per-agent badge, the Costs tab, and the top-bar month-to-date / today figures. Claude Code estimates cost client-side and it tends to run a little under the actual bill, so 1.1 nudges the numbers up ~10% to match. Display-only: recorded data is never changed, so resetting to 1 restores the raw figures. Empty or 1 = no adjustment. Also editable live on the Costs tab.
Show cost on subscription On a subscription there's no pay-per-token charge, so the Costs tab and per-agent cost badge are hidden by default. Enable this to show them in WHAT-IF mode — the estimated equivalent cost if you were billed pay-per-token (Claude Code's client-side total_cost_usd), clearly flagged as hypothetical. On pay-per-token the Costs tab always shows real spend regardless of this setting.

Activity bots — regular mode Each group header (and the top bar) shows a little row of robot icons — one per status present among its members, deduped: a bot dancing when an agent is working, standing still when idle, raising a ❓ when one awaits input, and so on. Especially handy on a folded group, where the per-agent rows are hidden — one glance tells you if anything's happening inside. Emoji bots are lightweight emoji+CSS; Pixel sprites are animated pixel-art robots; Off hides them. This is the plain dashboard's style (default Emoji bots). Stored as dashboard.activity_bots.regular.
Activity bots — slop mode 🎰 The same indicator's style when the dashboard's casino slop mode (the 🎰 in the header) is on — chosen independently of regular mode, the way slop already swaps the state pill for a slot machine. Default Pixel sprites (the full dancing robots), but you can keep the emoji bots, or turn them off. (Reduced-motion browsers get the bots without animation either way.) Stored as dashboard.activity_bots.slop.
Activity bots — wizard mode 🧙 The same indicator's style when the dashboard's wizard mode (the 🧙 in the header) is on — chosen independently of the other modes. Default Wizard glyphs (the fantasy 🧙🕯️📜💥 row). Switch to Pixel sprites to swap in the animated wizard spellcasters (casting when working, pondering when awaiting, a backfiring spell on error, an impatient staff-tap when idle) instead, or turn them Off. (Reduced-motion browsers get the bots without animation either way.) Stored as dashboard.activity_bots.wizard.
Plugins tab The Plugins tab manages human-defined integration bundles (e.g. a docker-backed MCP server). By default it auto-hides when you have no plugins installed — most setups never define one, so the empty tab is just clutter. Enable this to keep it visible regardless, e.g. to reach the built-in catalog and install one from there. (A broken plugins.json always shows the tab so the error isn't hidden.) Off by default. Stored as dashboard.always_show_plugins_tab.
Horizontal scroll On a window too narrow for a wide member table, the page scrolls sideways. The full-bleed bars (header, tabs, slop marquee) always stretch to fill the width so they never look ragged — this controls their content. Follow (default, checked) pins the header controls and tab strip to the viewport and makes them sticky-left, so they stay put and usable while you're scrolled right. Static (unchecked) lets that content scroll off with the page — the bars still fill the width, but the controls aren't reachable until you scroll back. Stored as dashboard.hscroll_follow (the default is omitted; static writes false).
Group quick options Each group header carries a row of editable "quick option" chips — 📝 description, 📁 default dir, 🧠 default profile, 🔗 links. They can grow the header wide. Checked (default) folds them to icon-only at rest and slides the text open when you hover the group header — a horizontal accordion that reclaims the space. The group name, activity bots and 👥 member count always stay visible. Unchecked keeps the full chips always shown (the original look). Touch devices (which can't hover) always see the full chips regardless. You can also pin a single group's chips open from its ⚙ menu (📌 pin quick options) so it stays expanded even with auto-fold on. Stored as dashboard.group_quick_options (the default hover is omitted; unchecked writes expanded).
Default terminal The per-agent focus (👁), open window and open terminal actions — plus the same commands in the ⌘ palette, a click on a CWD path cell, and a message's focus button — normally pop a native OS terminal window. Checked routes them all to an in-browser terminal pane in the dashboard's Terminals tab instead — the same surface the dedicated "web term" / "web window" buttons always use — so you never leave the browser. The dedicated web buttons (already always web) and the native-window bulk 🪟 windows… modal are unaffected. Off by default. Stored as dashboard.default_terminal (the default native is omitted; checked writes web).
Agent hide button Each agent row opens with two eye icons: focus (👁, raise the agent's terminal window) and hide (the slashed-eye, detach the window — the agent keeps running). Hiding a window is far less used than focusing it, so the dashboard drops the hide icon by default to keep the quick-control cluster tight — only focus shows. Checked brings the hide icon back for anyone who wants the row shortcut. Detaching is always possible from the terminal itself; this is only the per-row button. Off by default. Stored as dashboard.show_agent_hide_button (the default is omitted; checked writes true).
Group descriptions Every group can carry a short description — the 📝 chip beside the group name, click-to-edit. It's a display-only label: nothing in tclaude reads it, so it's a deprecated feature that mostly just clutters the header. The dashboard hides the chip by default; existing descriptions are kept in the database but not shown. Checked brings the chip back (and with it the only way to view or edit a group's description). Off by default. Stored as dashboard.show_group_description (the default is omitted; checked writes true).

ⓘ How deploying works — pattern, process & rhythms🔮 How a summoning works — rite, quest & drumbeats

Three things shape a deployed force and they work together: the work pattern briefs it once, the process gives it a shared map of phases to advance through, and the rhythms keep it moving between them.

  • Work pattern — the briefing. An ordered list of messages delivered once, after the whole team has spawned. Each step goes to one member (often the lead, who then fans the work out) or to everyone. It fires at deploy and does not repeat — but Re-brief re-delivers the template's current pattern to the live team on demand.
  • Process — the plan. An ordered map of phases the team advances through. It is advisory: advancing records a transition and nudges the roles now active — nothing is blocked, no permissions change, nothing auto-advances. A shared sense of where the work is, not a gate.
  • Rhythms — the pulse. Recurring nudges materialized as group cron jobs when you deploy — a snapshot taken then. Editing the template afterwards does not retune a force already in the field. Standing the force down deletes them; a retire that empties the group disables them (a later resume re-enables them).

Three powers shape a summoned party, woven together: the rite of command briefs it once, the quest plan hands it a shared map of chapters, and the drumbeats keep it marching between them.

  • Rite of command — the opening whispers. An ordered set of whispers spoken once, after the whole party has stood. Each verse reaches one familiar (often the lead, who parcels out the quest) or the whole party. It sounds once at the summoning and does not echo — though a re-brief speaks the circle's current rite to the living party again on command.
  • Quest plan — the chapters. An ordered map of chapters the party moves through. It is a map, not a wall: turning to the next chapter marks the passage and rouses the heroes it calls for — nothing is barred, no wards shift, no page turns itself. A shared sense of where the tale stands, not a locked door.
  • Drumbeats — the pulse. Recurring nudges struck as cron the moment the party is summoned — a cast taken then. Redrawing the circle later will not retune a party already afield. Standing the party down silences them for good; a retire that empties the ranks stills them (a later resume wakes them again).
